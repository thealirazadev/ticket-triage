## Decisions log

- 2026-07-18: Background work is a single polling worker thread with the database as the queue (no broker, no BackgroundTasks) - restarts resume unprocessed tickets, webhook replays cannot double-enqueue, and the failure paths stay readable. Limits documented in `docs/architecture.md`.
- 2026-07-18: LLM output enters the system only through the strict `TriageResult` schema, with one repair call and then a `needs_human` fallback; label enums are additionally CHECK-constrained in the database.
- 2026-07-18: Eval baseline and dataset are committed repo files; the CI eval job is path-gated (prompt/classifier/client/evals changes) to bound provider spend.
- 2026-07-18: Commit granularity increased at owner request mid-build: each discrete change (schema, model, migration, endpoint, service, test file) is its own commit rather than one commit per phase feature. `main.py` and the tickets router were built up incrementally so every commit stays importable.
- 2026-07-18: Timestamps serialize as ISO-8601 with a trailing `Z` via a `PlainSerializer` annotated type; SQLite returns naive datetimes, so a missing tzinfo is treated as UTC.
- 2026-07-18: The email-webhook `from` -> `sender` mapping uses a `model_validator(mode="before")` instead of a pydantic field alias; FastAPI re-applies field aliases in a way that emits a spurious `UnsupportedFieldAttributeWarning`.
- 2026-07-18: One `llm_calls` row per provider call. `complete()` records `ok`/`timeout`/`api_error`; the classifier flips a recorded `ok` row to `parse_error` when 200-response content fails `TriageResult` validation, so the parse-failed call keeps a single accurate row.
- 2026-07-22: A provider 200 with an undecodable body is treated as a retryable `api_error` rather than a parse failure. The envelope is a transport concern (a proxy HTML error page or a truncated body), distinct from valid-envelope-but-invalid-triage, which stays on the repair path. This keeps the breaker meaningful: a provider returning garbage 200s now trips it instead of silently looping.
- 2026-07-22: Non-string `message.content` is coerced to text in the client rather than guarded in the classifier, so `LlmResult.content: str` is true at runtime and the single parse gate stays the only place that interprets provider text.
- 2026-07-22: Benchmark reports two storage rows. Disk measures the real durable path (one commit fsync per ticket) and is storage-bound; tmpfs runs identical code and migrations in RAM to isolate the pipeline's CPU cost. Reporting only one number would misattribute disk latency to the pipeline, or vice versa.
- 2026-07-22: `docs/architecture.md` now records FastAPI 0.133 (Starlette 1.x), pytest 9, and Black 26 instead of FastAPI 0.115 / pytest 8. The stack choices did not change, only their floors: the six Starlette advisories are fixed in 1.3.1, and FastAPI pins Starlette with an upper bound in every release before 0.133.0, so the documented FastAPI version is a security floor rather than a preference. Upgrade target for each package is the lowest patched version, not the newest release, so the diff stays small enough to attribute any regression.
- 2026-07-22: Starlette 1.x deprecates `httpx` under `starlette.testclient` in favour of `httpx2`. Left alone: the warning is a `UserWarning` subclass so it does not trip `filterwarnings = ["error::DeprecationWarning"]`, and switching would add a dependency for no security benefit while the app's own provider client stays on `httpx` (and on `httpx.MockTransport` in tests).
